🔄 Circular Dependency — The 3-Level Cache SENIOR INTERVIEW

🎯 Most Asked Senior-Level Question

"How does Spring resolve circular dependencies?"

Spring uses a three-level singleton cache inside DefaultSingletonBeanRegistry:

Level 1
singletonObjects
Fully created beans
← All deps injected
Level 2
earlySingletonObjects
Semi-created beans
← Exposed before full init
Level 3
singletonFactories
Object factories
← Lazy proxy resolution
Bean A starts creation
↓ needs Bean B
Bean B starts creation
↓ needs Bean A
Level 3 exposes Bean A as early reference (ObjectFactory)
↓
Bean B receives early Bean A → completes
↓
Bean A receives fully created Bean B → completes
Key Point Constructor injection CANNOT resolve circular deps. Only field/setter injection works because Spring can expose the early reference before constructor completes.

Dependency Injection — Full Detail

📦 Constructor Injection BEST PRACTICE

UserService.java✓ Recommended
public class UserService {
    private final UserRepository repo;

    public UserService(UserRepository repo) {
        this.repo = repo; // Dependency from outside
    }
}
Immutable — final prevents change
Testable — new UserService(mockRepo)
Circular detection — fails at startup
Mandatory — can't create without deps

⚔️ Field vs Setter vs Constructor

❌ Field Injection

@Autowired private UserRepository repo;
  • Hidden dependency
  • Reflection-based (slower)
  • No immutability
  • Hard to unit test

⚠️ Setter Injection

@Autowired public void setRepo(UserRepository r) { this.repo = r; }
  • Optional deps
  • Mutable
  • Can cause NPE if called out of order

✅ Constructor Injection

private final UserRepository repo;
  • Explicit deps
  • Immutable
  • Easy mocking
  • Plain Java (no reflection)
  • Spring team recommends

Spring Bean Lifecycle — Full Breakdown

🎯 Senior Interview Essential

Know EVERY step from class loading to destruction. The lifecycle has 12 distinct phases.

🔄 Complete Lifecycle — 12 Phases

Phase 1 — Constructor called (reflection instantiation)
↓
Phase 2 — Populate properties (DI injection — field/setter)
↓
Phase 3 — setBeanName() — BeanNameAware
↓
Phase 4 — setBeanClassLoader() — BeanClassLoaderAware
↓
Phase 5 — setBeanFactory() — BeanFactoryAware
↓
Phase 6 — setApplicationContext() — ApplicationContextAware
↓
Phase 7 — BeanPostProcessor#postProcessBeforeInitialization
↓
Phase 8 — @PostConstruct / InitializingBean#afterPropertiesSet / @Bean(initMethod)
↓
Phase 9 — BeanPostProcessor#postProcessAfterInitialization ← AOP PROXIES CREATED HERE
↓
Phase 10 — 🎉 Bean is fully ready (proxied if AOP needed)
↓ Container shutdown
Phase 11 — @PreDestroy / DisposableBean#destroy
↓
Phase 12 — Custom destroy method (@Bean(destroyMethod))
🔬 Critical — Phase 9 Timing

AOP proxies are created in Phase 9, AFTER @PostConstruct (Phase 8). So inside @PostConstruct, this refers to the raw bean, not the proxy! Any @Transactional method called from @PostConstruct will NOT have transaction support.

⚠️ Scoped Proxy — Injecting Shorter Beans into Singletons

Request-scoped bean in singletonproxyMode is critical
@Component
@Scope(value = WebApplicationContext.SCOPE_REQUEST,
       proxyMode = ScopedProxyMode.TARGET_PROXY)
public class RequestScopedBean { }

@Service
public class SingletonService {
    @Autowired
    private RequestScopedBean bean; // Actually a proxy!
}
🔬 How It Works

Spring injects a proxy that delegates to the real bean on every method call. The proxy resolves the actual bean from the request context at runtime. Without proxyMode, you'd get an error or a stale reference.

⚠️ Singleton + Mutable State = Race Condition

❌ Dangerous — Mutable Singleton

@Service
public class CounterService {
    private int count = 0;
    public void increment() { count++; }
}

Thread 1 reads 0, writes 1. Thread 2 reads 0, writes 1. ❌ LOST UPDATE!

✅ Stateless — Thread-Safe

@Service
public class UserService {
    private final UserRepository repo;
    // No mutable state → thread-safe by design
}

Stateless = inherently thread-safe.

🎯 Senior-Level Answer

"Spring beans are singleton, but Spring itself is thread-safe because beans are expected to be stateless. Mutable state in singletons requires explicit synchronization."

🔄 Proxy Types — JDK vs CGLIB

JDK Dynamic Proxy

Used when target implements an interface.

  • Uses java.lang.reflect.Proxy
  • Proxies only interface methods
  • Faster startup

CGLIB Proxy

Used when no interface exists. Spring Boot 2+ default.

  • Bytecode generation (subclass)
  • Cannot proxy final classes/methods
  • final methods = no AOP!

🚨 Self-Invocation — The #1 AOP Trap

Why @Transactional sometimes failsProxy bypass
@Service
public class PaymentService {
    public void processPayment() {
            updateLedger(); // this.updateLedger() — bypasses proxy!
    }

    @Transactional
    public void updateLedger() { }
    // Called from processPayment → NO transaction!
}
💡 Solutions
  • Self-inject: @Autowired private PaymentService self; then self.updateLedger()
  • AopContext: ((PaymentService) AopContext.currentProxy()).updateLedger() (requires @EnableAspectJAutoProxy(exposeProxy=true))
  • Separate service: Extract @Transactional methods to a different @Service

@Transactional is NOT magic. It is AOP + Proxy + TransactionManager. Nothing more.

🔄 Rollback Rules #1 Gotcha

Exception TypeDefaultWhy?Override
RuntimeException✅ RollbackUnexpected error—
Error✅ RollbackJVM-level failure—
Checked Exception❌ NO rollbackExpected by designrollbackFor=Exception.class
Common MistakeChecked exception — no rollback!
@Transactional
public void transfer() throws Exception {
    debit();
    throw new Exception(); // Checked → NO rollback! Debit not rolled back!
}
// Fix:
@Transactional(rollbackFor = Exception.class)

📊 Isolation Levels — Explained with Examples

LevelDirty ReadNon-RepeatablePhantomPerformanceDB Default
READ_UNCOMMITTED⚠️ Allowed⚠️⚠️Fastest—
READ_COMMITTED✅ Prevented⚠️ Possible⚠️ PossibleGoodPostgres, Oracle
REPEATABLE_READ✅✅ Prevented⚠️ PossibleModerateMySQL
SERIALIZABLE✅✅✅ PreventedSlowest ⚠️—
🎯 Explain These

Dirty Read: Tx A reads uncommitted data from Tx B. If B rolls back, A read garbage.

Non-Repeatable Read: Tx A reads row twice, gets different values (Tx B modified & committed).

Phantom Read: Tx A runs same query twice, gets different row count (Tx B inserted/deleted).

📋 Propagation — REQUIRES_NEW Pool Killer

PropagationExisting TX?Behavior⚠️ Risk
REQUIREDY/NJoin / CreateSafe (default)
REQUIRES_NEWYSuspend current, create NEWPool exhaustion!
NESTEDYJDBC savepointOnly JDBC, not JTA

⚠️ REQUIRES_NEW = +1 Connection

Each REQUIRES_NEW grabs a separate DB connection. With 10 threads × 2 connections = 20 total. Pool size 10 → deadlock.

🧪 TransactionTemplate — Programmatic Control

TransactionTemplateWhen annotations fail
@Service
public class PaymentService {

⚠️ Long Transaction = Production Disaster

❌ BAD

@Transactional
public void processPayment() {
    callExternalAPI(); // 3s HTTP call inside TX!
    saveData();
}
  • Connection held for 3+ seconds
  • Pool exhaustion
  • DB lock contention

✅ GOOD

public void processPayment() {
    callExternalAPI(); // No TX
    saveData(); // Only this is @Transactional
}

"Keep transaction boundary SMALL." — Senior-level answer.

📋 readOnly=true — Optimizations

@Transactional(readOnly = true)Performance hint
@Transactional(readOnly = true)
public User findById(Long id) { return repo.findById(id).orElseThrow(); }

What it actually does: Sets FlushMode.NEVER on Hibernate session → skips dirty checking. Hints JDBC driver. Does NOT prevent writes.

Exception Handling — Production Patterns

❌ Bad Practice
try { process(); } catch (Exception e) { }

Empty catch = swallowed exception = debugging nightmare.

✅ Global @RestControllerAdvice

GlobalExceptionHandler.javaCentralized error handling
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(...) {
        return ResponseEntity.status(404)
            .body(new ErrorResponse("USER_404", "User not found"));
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<?> handleValidation(...) {
        List<FieldError> errors = ex.getBindingResult().getFieldErrors();
        return ResponseEntity.badRequest().body(errors);
    }

    @ExceptionHandler(Exception.class) // Catch-all
    public ResponseEntity<?> handleGeneral(...) {
        log.error("Unexpected error", ex); // ALWAYS log!
        return ResponseEntity.status(500)
            .body(new ErrorResponse("INT_500", "Internal error"));
    }
}

📋 ResponseEntityExceptionHandler — Spring's Base

Extended HandlerOverride Spring MVC defaults
@RestControllerAdvice
public class ExtendedHandler extends ResponseEntityExceptionHandler {
    @Override
    protected ResponseEntity<?> handleHttpMessageNotReadable(...) {
        return ResponseEntity.badRequest()
            .body(new ErrorResponse("INVALID_JSON", "Malformed JSON body"));
    }
    // Override handleMethodArgumentNotValid, handleHttpMediaType, etc.
}

📋 Custom Cross-Field Validator

@PasswordMatch annotationClass-level validation
@Target({ ElementType.TYPE })
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = PasswordMatchValidator.class)
public @interface PasswordMatch {
    String message() default "Passwords don't match";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

public class PasswordMatchValidator implements ConstraintValidator<PasswordMatch, RegisterRequest> {
    public boolean isValid(RegisterRequest req, ConstraintValidatorContext ctx) {
        return req.getPassword().equals(req.getConfirmPassword());
    }
}

@PasswordMatch // ← Class-level!
public class RegisterRequest {
    @NotBlank private String password;
    @NotBlank private String confirmPassword;
}

Different validation per operation

public class UserRequest {

The problem1 query for entities + N for relations

// Total: 1 (users) + N (orders per user) = N+1 queries!

"N+1 is NOT a database problem. It is an ORM fetching strategy problem."

⚡ Lazy vs Eager Loading

Common Interview Traps — What Gets You Rejected

🎯 These Are the Questions That Differentiate Senior from Junior

Below are the most common traps experienced interviewers use to probe depth of understanding.

🚩 Trap 1: "How does @Transactional actually work?"

❌ Weak Answer

"It makes the method transactional."

✅ Strong Answer

"It's AOP-based. Spring creates a proxy (JDK or CGLIB) that wraps the method. The TransactionInterceptor (Around advice) calls PlatformTransactionManager.getTransaction() before method execution, and commit/rollback after. If the method is called from within the same class (self-invocation), the proxy is bypassed and @Transactional has NO effect."

🚩 Trap 2: "How does Spring resolve circular dependencies?"

❌ Weak Answer

"It doesn't. You should avoid circular dependencies."

✅ Strong Answer

"Spring uses a three-level singleton cache: singletonObjects (fully created), earlySingletonObjects (semi-created), and singletonFactories (object factories). When Bean A depends on Bean B and vice versa, Spring creates A, places an early reference via ObjectFactory in level 3, uses that to inject into B, then B is fully created and injected into A. Constructor injection cannot resolve circular deps — only setter/field injection works."

🚩 Trap 3: "What happens if I put @Transactional on a private method?"

❌ Weak Answer

"It works."

✅ Strong Answer

"It does NOT work. CGLIB proxies can only override public methods (they create subclasses). JDK proxies can only proxy interface methods which are always public. Private methods are not proxied, so @Transactional is silently ignored."

🚩 Trap 4: "What is the difference between @Component, @Service, @Repository, @Controller?"

❌ Weak Answer

"They all register beans."

✅ Strong Answer

"@Component is the generic stereotype. @Service adds no extra behavior — it's semantic (business logic layer). @Repository enables translation of persistence exceptions into Spring's DataAccessException hierarchy. @Controller enables component scanning for @RequestMapping annotations and is detected by HandlerMapping."

🚩 Trap 5: "Is Spring Boot's default scope singleton thread-safe?"

❌ Weak Answer

"Yes, singleton beans are thread-safe."

✅ Strong Answer

"Singleton beans are thread-safe ONLY if they are stateless. If you have mutable instance variables (like private int count), you WILL get race conditions because the singleton is shared across threads. Spring's container itself is thread-safe because beans are expected to be stateless — any mutable state must be explicitly synchronized by the developer."

🚩 Trap 6: "What is Open Session in View? Is it good?"

❌ Weak Answer

"It prevents LazyInitializationException, so it's helpful."

✅ Strong Answer

"OSIV keeps the Hibernate session open during view rendering. The DB connection is held throughout the entire HTTP request. This can cause connection pool exhaustion, long-running transactions, and hidden N+1 queries in views. It's enabled by default in Spring Boot (spring.jpa.open-in-view=true). The senior approach is to disable it and use DTO projections / fetch joins / EntityGraph instead."

🚩 Trap 7: "How does @Autowired resolve when there are multiple beans of the same type?"

❌ Weak Answer

"It throws an error."

✅ Strong Answer

"@Autowired first tries by type. If multiple candidates exist: (1) Check @Primary — if one bean has it, that wins. (2) Check @Qualifier — if specified, narrows by bean name. (3) Check parameter/field name — if it matches a bean name, that bean is used. If still ambiguous, NoUniqueBeanDefinitionException is thrown."

🚩 Trap 8: "What's the difference between REQUIRES_NEW and NESTED propagation?"

❌ Weak Answer

"They both create new transactions."

✅ Strong Answer

"REQUIRES_NEW SUSPENDS the current transaction and creates a completely NEW one — using a separate DB connection. If the new tx rolls back, the outer tx can still commit. NESTED uses a SAVEPOINT within the same transaction (JDBC savepoints). If the nested tx rolls back, only the savepoint is rolled back, not the entire outer tx. The outer tx can still commit. NESTED does NOT require a separate connection — it's just a savepoint."

🚩 Trap 9: "Can you use @Async with @Transactional?"

❌ Weak Answer

"Yes, they work together."

✅ Strong Answer

"They CAN be combined, but the transaction runs in the ASYNC thread, NOT the caller's thread. The caller's transaction does NOT propagate to the async method. Each @Async method with @Transactional creates its own transaction on a new thread. This is critical to understand — the connection is bound to the async thread, not the request thread."

🚩 Trap 10: "How does Spring Boot auto-configuration work?"

❌ Weak Answer

"It just works. Spring Boot configures things automatically."

✅ Strong Answer

"@EnableAutoConfiguration imports AutoConfigurationImportSelector which uses SpringFactoriesLoader to load META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports. This file lists ~130 auto-configuration classes. Each class has @Conditional* annotations (like @ConditionalOnClass, @ConditionalOnMissingBean, @ConditionalOnProperty) that determine whether the configuration applies based on classpath contents, existing beans, and configured properties. Auto-configuration is NOT magic — it's conditional bean registration."

Real Production Scenarios — Debugging Like a Senior

⚡ These Are Real Incidents I've Seen in Production

Understanding these scenarios separates engineers who can code from engineers who can solve real problems.

📋 Scenario 1: Connection Pool Exhaustion

⚠️ The Problem

Symptoms: App becomes unresponsive after running for a few hours. HikariPool-1 - Connection is not available, request timed out after 30000ms.

Root causes:

  • Long-running @Transactional methods (holding connections during HTTP calls)
  • REQUIRES_NEW grabbing multiple connections per request
  • Open Session In View holding connections during view rendering
  • Missing connection release in @Async methods
  • Leaked connections from third-party libraries

Diagnosis: Enable Hikari metrics: spring.datasource.hikari.metrics=true. Monitor active vs idle connections. Add thread dump analysis.

Fix: Move external API calls outside transactions. Reduce REQUIRES_NEW usage. Set spring.jpa.open-in-view=false. Right-size pool: spring.datasource.hikari.maximum-pool-size=20.

📋 Scenario 2: The Silent N+1 That Kills Performance

⚠️ The Problem

Symptoms: API endpoint that lists 100 users takes 5+ seconds. DB CPU spikes. Only 100 users, so pagination is not the issue.

Root cause: N+1 query problem. findAll() executes 1 query for users + 100 queries for each user's orders = 101 queries.

Diagnosis: Enable Hibernate stats: spring.jpa.properties.hibernate.generate_statistics=true. Look for "100 statements" logged for a single endpoint. Or use datasource-proxy to log all queries.

Fix: Use JOIN FETCH, @EntityGraph, or DTO projection. Always check query count before deploying to production.

📋 Scenario 3: Transactional Method Called from Within Same Class

⚠️ The Problem

Symptoms: A @Transactional method DOES roll back when called directly from a controller, but DOES NOT roll back when called from another method in the same service.

Root cause: Self-invocation problem. this.method() bypasses the AOP proxy.

Diagnosis: Check if the caller is in the same class. If service.methodA() calls this.methodB() where B is @Transactional, the proxy is bypassed.

Fix: Inject the proxy into itself (@Autowired private MyService self), use AopContext.currentProxy(), or extract the transactional method to a separate service.

📋 Scenario 4: LazyInitializationException in REST Controller

⚠️ The Problem

Symptoms: LazyInitializationException: could not initialize proxy — no Session when accessing user.getOrders() in a controller or serializer.

Root cause: The Hibernate session was closed before the lazy association was accessed. The entity was loaded, the transaction committed, session closed, then Jackson serialization triggered lazy loading.

Fix: Use DTO projections (best), JOIN FETCH in the query, @EntityGraph, or disable OSIV only after you've fixed all lazy loading issues.

📋 Scenario 5: Cache Stampede Takes Down Database

⚠️ The Problem

Symptoms: Every hour at :00, DB CPU spikes to 100%. Queries time out. Cache TTL is set to 1 hour.

Root cause: Cache stampede — all cache entries expire at the same time, and all concurrent requests hit the database simultaneously.

Fix: Add random jitter to TTL (entryTtl(Duration.ofMinutes(60 + Random.nextInt(10)))), use @Cacheable(sync=true), or pre-warm the cache before the stampede window.

📋 Scenario 6: Slow Spring Boot Startup in Production

⚠️ The Problem

Symptoms: 60+ second startup time. Kubernetes liveness probe kills the pod before it's ready.

Root causes & fixes:

  • Classpath scanning: Limit @ComponentScan to specific packages
  • Auto-configuration lag: Exclude unused auto-configs with @SpringBootApplication(exclude=...)
  • JPA entity scanning: Set spring.jpa.properties.hibernate.archive.autodetection=none
  • Lazy init: spring.main.lazy-initialization=true
  • AOT compilation: Use Spring Boot 3+ AOT for image optimization

📋 Scenario 7: Security Misconfiguration — Exposed Internal APIs

⚠️ The Problem

Symptoms: A new developer adds a controller without realizing security isn't applied globally. The endpoint /api/admin/reset-password is accessible without authentication.

Fix: Use deny-by-default: .anyRequest().authenticated() instead of permit-all. Use @PreAuthorize on all admin methods. Never rely on developers remembering to secure endpoints — use centralized security rules in SecurityFilterChain. Run security audit tests that verify all endpoints require auth.

📋 Scenario 8: Async Method Exceptions Are Silently Swallowed

⚠️ The Problem

Symptoms: An @Async method fails but nobody notices. The caller continues as if nothing happened.

Root cause: @Async methods run in a separate thread. If the method returns void, exceptions are caught by the AsyncUncaughtExceptionHandler and typically logged but NOT propagated. If the method returns Future/CompletableFuture, the exception is captured in the Future.

Fix: Always return CompletableFuture from async methods. Configure a custom AsyncUncaughtExceptionHandler to alert on failures. Never use void for async methods that can fail.
